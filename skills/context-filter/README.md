# context-filter

一个用于实现“干净交付（Clean Delivery）”与“无偏开始（Unbiased Start）”的元技能（Meta-Skill）。

## 什么是 Context Filter？

在 Agent 的迭代开发与长期会话中，上下文不可避免地会积累大量“噪声”：试错的陈旧路径、特定本地环境的临时约束、过度具体的负面提示词（如“不要做 X”）等。这些噪声在交付给下一个 Agent 或对外发布文档时，会产生负面偏见，迫使后续模型沿着已被否定的路径重复思考。

`context-filter` 是一种信息降噪与状态解耦模式：

```
Polluted Context/      # 历史污染上下文：包含试错路径、临时环境变体、偶发性负面提示词
       │
       ▼ [Context Filter] (识别真正的 Baseline 与核心信号)
       │
  Clean State/         # 干净状态：脱离具体情境的抽象文档 / 无偏提示词
    ├── Signal         # 核心领域逻辑、基础硬性约束（架构、不可变限制）
    └── Noise (Filtered)# 移除：运行时变体、已被解决的 corner case、开发偶发性噪声
```

- **无偏开始 (Unbiased Start)**：从全新的、未被污染的上下文中启动任务或派发 subagent，防止前序会话的偏见与噪声干扰其决策，使其拥有更强的泛化能力。
- **干净交付 (Clean Delivery)**：交付一份抽离了开发历史细节、处于“稳态”的纯净成果文档，确保后续使用者、维护者或新 Agent 看到的上下文是清晰且一致的。

## 隐喻思路：信号与噪声 (Signal vs. Noise)

这个技能的核心隐喻源自**信号处理（Signal Processing）**。在复杂的上下文中，我们需要具备识别“真正基线（Baseline）”、“真正信号（Signal）”以及“真正噪声（Noise）”的能力：

1. **真正的信号 (Signal)**：是系统的核心领域逻辑与硬性约束（例如某些不可变的 API 限制、根本性的架构决定）。这些内容决定了系统的边界，必须被准确保留。
2. **真正的噪声 (Noise)**：是运行时的偶然变体（Runtime Variables）、开发过程中的试错历史，或者是特定本地开发环境下的临时妥协（例如仅在某个开发会话中临时出现的偶发性偶合）。
3. **稳态与负面提示词的去留**：
   - *必须保留的负面提示*：如果系统尚未完全收敛到稳态，或者新 Agent 在没有显式说明的情况下，会极其自然且错误地走向某条死胡同（如由于底层限制必须调用 A 端口而非 B 端口），此类约束属于核心信号，应予保留。
   - *必须清除的负面提示*：如果是开发过程中由于偶然因素产生的“不要如何”的警告，或者随着版本迭代该 corner case 在底层已被化解，那么继续保留它反而会消耗模型的注意力，甚至诱导模型产生多余的反向思考。

通过将繁杂的上下文降噪，提炼出独立于具体演进情境的抽象文档，就能在去除噪声的同时，实现与原始冗长上下文相近甚至更优的泛化执行效果。

## 使用方法 (Usage)

当需要为新 Agent 构造干净启动提示词，或者需要将当前充满试错细节的文档整理成持久化文档库时，请调用此技能：

1. **分析 (Analyze)**：审查目标上下文或文档，映射出绝对规则与历史演进旅程。
2. **解耦 (Decouple)**：将普适性的架构需求与局部的情境历史剥离。
3. **抽象重写 (Abstract & Rewrite)**：从更高层的客观视角重写目标内容。围绕“系统如何运作”以及“应当做什么”进行正面构建，仅在面临硬性底层限制时使用负面提示。
4. **交付 (Deliver)**：提供无偏的纯净成果，不向后续流程传递不必要的认知负担。
